# 70 范围、边界与目标形态

## 一、本文要回答什么

本文件回答四个问题：

1. `docs/70-sandbox/` 到底在做什么，不在做什么
2. 当前系统允许讨论哪些领导问题，不能越界讨论哪些问题
3. 现有项目库与图谱数据分别能强支撑什么、弱支撑什么、暂时支撑不了什么
4. 最终系统的目标形态是什么，哪些对象是业务本体，哪些只是分析层宽表

这里讨论的是完整最终形态下的 `sandbox` 总体边界，不是某个临时页面、问答入口或调试 demo 的边界。

## 二、系统定位

`sandbox` 是领导视角的将来时研判系统，不是：

- 历史报表系统
- 通用问答系统
- 把图表和大模型硬拼在一起的 toy dashboard

系统统一承载两类能力：

1. `trend = baseline world forecast`
   在当前政策规则不显著变化的前提下，判断未来 1-2 个年度周期内，主题、机构、人员、项目和资金结构会怎样演化。
2. `simulation = counterfactual policy scenario`
   在 baseline world 上施加结构化政策情景，比较“如果这样调政策”与“如果什么都不改”之间会出现什么差异。

因此，本目录的核心不是“展示历史发生了什么”，而是：

- 发现哪些变化已经进入政策视野
- 判断这些变化在不干预下会如何延续
- 比较不同政策方案可能带来的结构后果
- 为领导形成可追溯、可解释、可比较的决策依据

## 三、纳入范围的核心问题

系统纳入的，不是所有与科技管理有关的问题，而是与当前数据底座和政策推演任务直接相关的问题。

### 1. 趋势预判纳入的问题

- 哪些主题正在持续升温、回落或发生结构迁移
- 哪些主题存在“申报热度上升，但评审进入、立项或合同兑现没有同步改善”的背离
- 哪些主题出现机构过度集中、协作减弱、主体更新不足、评分与资源配置错配等结构风险
- 在不新增政策动作的前提下，未来 1-2 个年度周期的 baseline world 将朝哪个方向继续演化

### 2. 沙盘推演纳入的问题

- 如果调整指南范围、准入条件、预算分配、评审偏好、主题配额或协作导向，未来结构会怎样变化
- 在预算、容量和风险约束不变时，哪些主题受益，哪些主题被挤出，代价落在什么地方
- 不同政策方案分别改善了什么，又分别引入了什么副作用
- 哪种政策组合更接近领导当前的治理目标，例如“控过热、稳重点、促协作、降集中”

### 3. 交付层纳入的问题

- 如何把 trend 和 simulation 的结果组织成领导可读的结论
- 如何让领导追问每条结论背后的事实来源、口径、样本和假设
- 如何把领导自然语言意图稳定映射成可执行的政策情景输入

## 四、不纳入范围的内容

以下内容可以存在于系统外围，但不属于 `docs/70-sandbox/` 所定义的分析本体：

- 单纯的历史统计报表
- 脱离证据链的自由问答
- 先画图后补定义的可视化页面
- 把任意参数调节包装成“政策推演”
- 不披露假设来源的自动简报生成

更具体地说，以下做法在本总纲下都不成立：

- 把 `topic × year` 或 `topic_time_panel` 当成整个业务本体
- 把一个“热点排行榜”页面误当成趋势预判
- 把若干临时参数集合误当成正式政策输入
- 把图谱结构指标直接说成真实业务结果
- 把 LLM 生成的语言输出误当成分析结果本身

## 五、业务本体与分析层的边界

本系统必须严格区分“业务本体”和“分析层宽表”。

### 1. 业务本体

业务本体必须由三层组成：

- `事实层`
  围绕 `Project` 生命周期展开的真实业务事实，包括申报、评审、立项、合同、经费等事件。
- `关系层`
  围绕 `Topic / Institution / Person / Project` 的主题、机构、人员和项目关系网络。
- `统计层`
  为趋势识别、比较与推演服务的统计面板和派生指标。

### 2. 分析层宽表

`topic × year`、`topic × institution × year`、`topic × person × year` 这类宽表只属于统计层或 analysis mart。

它们的作用是：

- 承接聚合计算
- 支撑趋势识别
- 作为推演状态层的输入视图

它们不能承担的角色是：

- 代替项目生命周期事实
- 代替主题、机构、人员关系网络
- 直接充当整个系统的业务本体

换句话说：

`topic × year` 很重要，但它只是分析层宽表，不是系统世界模型。

## 六、当前数据底座

当前系统能够使用的真实数据，主要来自两部分。

### 1. 项目库

当前明确可用的主链是：

`Sb_Jbxx -> Sb_Sbzt -> Sb_Jfgs -> PS_XMPSXX -> Ht_XMLXXX -> Ht_Jbxx -> Ht_Jfgs`

这条链可以支持系统围绕同一项目主键观察：

- 申报事实
- 评审过程与评分结果
- 立项事实
- 合同事实
- 申报经费与合同经费

### 2. 图谱

图谱提供的是结构增强能力，而不是替代业务事实。

当前可作为结构信息来源的对象包括：

- `Project`
- `Topic`
- `Institution`
- `Person`
- 与上述对象相关的主题连接、协作连接、人员参与连接等关系

图谱能支撑：

- 主题关联与迁移分析
- 机构协作结构分析
- 人员桥接与结构位置分析
- 主题、机构、人员在网络中的结构特征提取

但图谱默认仍然属于“关系层 + 结构代理”，不是最终业务结果表。

## 七、当前数据支撑强弱

系统必须先承认数据边界，再决定能说什么。

### 1. 强支撑

以下问题，当前数据可以作为领导级研判的硬底座：

| 范围 | 当前支撑 | 说明 |
|---|---|---|
| 申报规模与主题热度 | 强 | 可基于 `Sb_Jbxx / Sb_Sbzt / Sb_Jfgs` 稳定统计 |
| 评审进入、评分、排序代理 | 强 | 可基于 `PS_XMPSXX` 构建评审层状态与质量代理 |
| 立项与合同事实 | 强 | 可基于 `Ht_XMLXXX / Ht_Jbxx / Ht_Jfgs` 识别立项、合同与合同经费 |
| 主题年度结构变化 | 强 | 可稳定构建 `topic × year` 等 analysis mart |
| 机构参与与集中度 | 强 | 可基于项目-机构关系形成活跃度、集中度等指标 |
| 预算重分配、配额调整、评审偏好变化的反事实比较 | 强到中 | 前提是以项目事实和评审排序为约束，而不是拍脑袋改结果 |

### 2. 中支撑

以下问题可以进入系统，但必须明确是“基于代理或规则假设”的结果：

| 范围 | 当前支撑 | 说明 |
|---|---|---|
| 主题迁移强度 | 中 | 需要依赖图谱关系和跨年结构变化做二次计算 |
| 协作能力变化 | 中 | 可由项目协作网络和图谱结构代理近似，不等于真实组织能力 |
| 人员接续风险 | 中到弱 | 只有在人员数据覆盖足够时才能形成风险信号，不能直接下人才断层结论 |
| 评审偏好调整后的结构影响 | 中 | 可做规则驱动反事实，但不应包装成真实因果识别 |
| 协作激励类政策 | 中到弱 | 需要较强结构假设，适合作为条件性推演，不适合作为确定结论 |

### 3. 弱支撑或当前不支撑

以下问题当前不能直接作为系统硬结论：

| 范围 | 当前支撑 | 原因 |
|---|---|---|
| 真实验收率 | 弱或不支撑 | 当前链路未形成稳定、完备、可审计的验收事实口径 |
| 真实转化率 | 弱或不支撑 | 缺少稳定、闭环、可验证的转化事实表 |
| ROI 或投入产出效率 | 不支撑 | 缺少完整收益侧结果数据 |
| 全省人才断层 | 不支撑 | 当前系统只掌握项目与关系数据，不掌握完整人才供给与流动底座 |
| 外部产业、舆情、市场冲击 | 不支撑 | 当前未接入系统外部数据源 |
| 已被证明的强因果结论 | 不支撑 | 当前系统不是因果识别平台，最多提供条件性结构推演 |

## 八、系统输出边界

基于以上数据条件，系统允许输出三类内容：

### 1. 事实层输出

- 主题、项目、机构、人员、经费的已观测变化
- 项目生命周期中可直接验证的申报、评审、立项、合同事实
- 当前组合结构、集中度、分布与变化方向

### 2. 解释层输出

- 对热点升温、结构背离、协作减弱、集中度上升等现象的机制解释
- 对某些主题为何被判为高热、脆弱、拥挤或风险累积的规则化说明
- 对模型使用了哪些代理量、哪些结构特征的披露说明

### 3. 推演层输出

- baseline world 的未来判断
- baseline 与不同政策情景之间的差异比较
- 受益方、受损方、挤出效应和风险迁移
- 基于明确假设和约束的政策建议

系统不应直接输出以下内容：

- 不披露假设的确定性预测
- 把条件性反事实结果写成现实世界必然结果
- 把相关性、排序或结构相似性说成已识别的因果关系
- 把代理变量说成真实验收、真实转化或真实人才供给结论

## 九、LLM 的职责边界

LLM 是系统中的辅助能力，不是事实来源，也不是替代计算引擎。

允许 LLM 承担的职责：

- 解析领导问题
- 将政策语言映射成结构化 `Scenario Contract`
- 归纳多指标、多证据结果
- 生成领导简报与追问回答
- 将证据链组织成可读表达

不允许 LLM 承担的职责：

- 编造缺失数据
- 自行决定核心业务指标
- 跳过规则与模型直接给出政策效果
- 在未披露证据等级的情况下输出强判断

## 十、最终目标架构

最终目标不是做两个分裂的小模块，而是在统一 `sandbox` 下形成一套完整的政策研判体系。

目标架构应为：

```text
项目库事实 + 图谱关系
  -> 统一对象模型（Project / Topic / Institution / Person / Review / Funding / Contract）
  -> 统一统计层与 analysis mart
  -> Trend：形成 baseline world forecast
  -> Simulation：在 baseline world 上运行 counterfactual policy scenario
  -> Evidence / Briefing / QA：面向领导展示、解释、追问
```

其中：

- `trend` 是基线世界推断引擎
- `simulation` 是政策情景反事实引擎
- `briefing / qa` 是消费分析结果的交付层，不是分析本体

系统最终要做到的，不是“把过去讲明白”，而是：

- 用真实项目事实识别未来值得治理的问题
- 用统一世界模型比较不同政策路径的后果
- 用可追溯证据把领导结论站稳

## 十一、范围控制原则

后续任何设计、编码、页面或接口，只要落在 `docs/70-sandbox/` 下，都必须同时满足以下约束：

1. 必须围绕 `trend` 或 `simulation` 之一服务，不能回到泛报表逻辑
2. 必须建立在统一对象模型之上，不能拿 analysis mart 充当业务本体
3. 必须标明观测、代理、结构假设或不支撑，不能混说
4. 必须能够回溯到项目事实、聚合口径和规则链
5. 必须服务于领导视角的将来时研判，而不是单纯复述过去

如果不满足这五条，就不属于本目录定义的 `sandbox` 正式范围。
