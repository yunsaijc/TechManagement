# 70 面向最终形态的实施顺序

## 一、本文要解决什么

本文件定义的不是“先做一个小版本、再慢慢补”的产品分期叙事，而是完整 `sandbox` 目标形态下的实施依赖顺序。

这里的“顺序”只回答一个问题：

为了最终形成领导视角的将来时研判系统，哪些东西必须先被锁定，哪些东西可以并行，哪些东西绝不能倒过来做。

如果顺序错了，结果通常只有两种：

- 做成报表系统再硬加推演
- 做成问答外壳再回头补事实和证据链

这两种都会把系统重新拉回 toy。

## 二、总原则

`Sandbox` 的实施顺序必须服务于最终完整架构：

`真实数据 -> 统一对象模型 -> 统一统计口径 -> baseline world -> counterfactual scenario -> evidence / briefing / qa`

这条顺序表达的是“依赖关系”，不是“先只做前面的简化版”。

任何把交付层放到分析骨架前面的做法，都应该被视为顺序错误，例如：

- 先堆图表页面，再定义世界模型
- 先做自然语言问答，再定义政策输入合同
- 先画政策效果图，再定义状态转移链
- 先发明若干参数，再回头找数据解释含义

## 三、必须先锁定的系统骨架

### 1. 先锁定统一对象模型

必须先回答“系统里到底有哪些对象，主键是什么，彼此如何关联”。

至少要锁定：

- `Project` 如何锚定到 `Sb_Jbxx.id`
- `Topic` 如何由 `zndm / guide_name / zxmc` 等口径稳定映射
- `Institution` 如何规范名与去重
- `Person` 如何区分负责人、参与人和图谱中的同名异人问题
- `Review / Funding / Contract` 是静态属性还是生命周期事件

如果这一步不先锁，后面的 trend、simulation、LLM 解释都会漂移，因为它们消费的根本不是同一个世界。

### 2. 再锁定时间轴与生命周期

在对象模型之后，必须立即锁定“变化”到底沿哪条时间轴被观测和比较。

至少要统一：

- `application_year`
- `review_year`
- `award_year`
- `contract_year`
- `event_year`
- `application_cohort_year`

还要明确：

- 哪些问题必须用 cohort 视角看
- 哪些问题可以按 event year 直接看
- 哪些指标跨年比较时必须做对齐

如果时间轴不先锁，所有“趋势”“改善”“恶化”“兑现”“背离”都会失真。

### 3. 再锁定指标口径与证据等级

在对象和时间轴之后，必须写清楚系统能稳定算什么，哪些只是代理，哪些根本不能说。

至少要形成以下统一规则：

- 指标定义
- 聚合口径
- 对象粒度
- 时间口径
- `Observed / Proxy / Structural Assumption / Unsupported` 标签

这一步的意义是：

- 防止同一指标在不同页面、接口、分析链里口径漂移
- 防止把结构代理冒充真实结果
- 防止领导页面出现“看起来完整、实际上不可审计”的结论

### 4. 再锁定 analysis mart，而不是反过来

只有当前三步完成后，才应该去建设 `topic × year` 等分析层宽表。

必须明确：

- analysis mart 服务于分析，不定义业务本体
- analysis mart 来源于事实层和关系层
- analysis mart 只是 trend 和 simulation 的输入视图之一

因此，正确顺序是：

`先有世界模型，再有 analysis mart`

而不是：

`先有 topic_time_panel，再拿它倒推整个系统`

### 4.5 先形成 sandbox 共享数据层

在 analysis mart 真正稳定之前，代码层必须先形成一层共享数据映射。

原因很直接：

- `trend` 和 `simulation` 都要用同一条项目生命周期链
- 两者都要消费同一套 `Topic / Project / Review / Funding / Contract` 对象
- 如果不先共享，后面一定会出现两套 topic 口径、两套 funded 口径、两套经费口径

因此，代码结构上至少应先落：

- `src/services/sandbox/data/`

它负责：

- 统一项目库生命周期事实
- 统一 topic 映射
- 统一评审与经费口径
- 统一输出 `load_project_facts(...)`、`build_topic_aggregates(...)`、`build_topic_year_aggregates(...)`

这层先于 `trend` 预测逻辑，也先于 `simulation` 状态转移逻辑。

共享层的详细契约见：

- [08-shared-data-layer.md](/home/tdkx/workspace/tech/docs/70-sandbox/overview/08-shared-data-layer.md)

### 5. 再锁定 baseline world

在对象、时间轴、口径和 analysis mart 稳定之后，才能建立 `trend`。

`trend` 的职责不是复述历史，而是构建 baseline world：

- 当前结构状态是什么
- 哪些变化已经形成趋势信号
- 在当前政策规则不显著变化时，未来 1-2 个年度周期大概率沿什么方向继续演化

这一步必须先于 `simulation`，因为：

- 没有 baseline world，就没有反事实比较基准
- 没有 baseline world，所谓“政策效果”其实只是在直接改结果变量

### 6. 再锁定政策情景合同

在 `simulation` 开始之前，必须把政策输入写成正式的 `Scenario Contract`。

至少包含：

- `intent`
- `baseline_scope`
- `actions`
- `constraints`
- `evaluation`
- `validation`

重点不是字段形式，而是要先回答：

- 领导这次到底要优化什么
- 允许动哪些政策抓手
- 哪些抓手当前数据能支撑，哪些只能弱支撑
- 哪些目标如果数据不支持，就必须显式降级表达

如果这一步不先完成，`simulation` 就会退化成参数试玩器。

### 7. 最后锁定状态转移链与反事实引擎

`simulation` 真正要实现的不是“把某个指标调高调低”，而是定义政策如何沿治理流程作用于系统状态。

当前应该围绕的主链是：

`申报响应 -> 评审选择 -> 立项进入 -> 合同配置 -> 结构外溢`

其中每一段都要明确：

- 受哪些政策动作影响
- 需要哪些真实数据支撑
- 哪些部分依赖结构假设
- 结果最终沉淀到哪些状态变量

只有在这条链被定义清楚后，`simulation` 才是反事实政策情景，而不是直接改图表。

## 四、围绕最终形态的并行工作面

在系统骨架锁定后，下面这些工作可以并行推进，但不能越过前置依赖。

### 1. 数据与对象工作面

- 项目库字段清洗与生命周期映射
- 图谱对象映射与主键治理
- 主题、机构、人员规范口径整理

### 2. 统计与特征工作面

- `topic × year`
- `topic × institution × year`
- `topic × person × year`
- `project_cohort_outcome`
- 图谱结构代理指标

### 3. Trend 工作面

- baseline world 构建
- 趋势信号检测
- 风险清单生成
- 基线未来判断

### 4. Simulation 工作面

- 领导语言到 `Scenario Contract` 的映射
- 政策动作可执行化
- 状态转移链计算
- baseline 与 scenario 对比输出

### 5. 交付与 LLM 工作面

- 证据链组织
- 领导简报结构
- 多轮追问接口
- 边界披露与结论解释

并行的前提始终只有一个：

消费同一套对象模型、同一套时间轴、同一套指标口径、同一套证据标签。

## 五、面向最终形态的推荐实施顺序

下面给出的不是“旧 phase 叙事”，而是完整系统落地时不可颠倒的依赖顺序。

### 1. 先把“世界”定义清楚

优先完成：

- 对象模型
- 生命周期事件模型
- 主键与映射规则
- 时间轴规则

产物应当是统一世界模型，而不是几张聚合表。

### 2. 再把“能算什么、能说什么”定义清楚

优先完成：

- 指标总表
- 粒度规则
- 聚合规则
- 证据等级标签
- 当前不支撑结论清单

产物应当是统一口径，而不是先做几个页面再用语言补解释。

### 3. 再把 baseline world 建出来

优先完成：

- 当前结构快照
- 主题趋势信号
- 风险信号
- 基线未来判断

产物应当是一个可供 `simulation` 调用的 baseline world，而不是几张历史统计图。

### 4. 再把政策情景输入和作用链建出来

优先完成：

- `Scenario Contract`
- 政策动作分类与支撑等级
- 约束表达
- 评价目标表达
- 状态转移链

产物应当是可执行、可比较、可审计的政策情景，而不是随手调参。

### 5. 最后把领导交付层接上去

优先完成：

- 领导简报结构
- 证据链展示
- 多轮追问
- 边界披露

产物应当是消费 trend 与 simulation 结果的交付界面，而不是倒过来驱动分析定义。

## 六、哪些东西绝不能晚于实现

以下内容属于系统骨架，不能在代码已经铺开之后再慢慢补：

- 对象模型
- 时间轴规则
- 指标口径
- 证据等级
- baseline world 定义
- `Scenario Contract`
- 状态转移链

这些如果晚锁定，会直接造成三种后果：

- 文档、代码、页面各说各话
- 同一结论在不同地方用不同口径计算
- 领导看到的是“像结果的东西”，但背后无法追溯

## 七、哪些可以后接，但不能反向定义系统

以下内容可以在系统骨架稳定后逐步丰富，但不允许反过来定义分析本体：

- 页面布局与视觉风格
- 图表形态与交互细节
- 简报语言润色
- 通用问答包装

这些都属于交付层，不应成为系统设计的起点。

## 八、实施中的停工条件

如果出现以下情况，应该停下来补骨架，而不是继续写更多代码或页面：

1. `Topic` 口径仍然混用，无法稳定定位主题对象
2. cohort 与 event year 仍然没有切开
3. analysis mart 被继续当作业务本体使用
4. `trend` 和 `simulation` 再次混成同一条逻辑链
5. 政策输入又退回成若干临时参数集合
6. 任何领导结论无法回溯到项目事实、聚合口径和规则链

这六种情况任意出现，继续推进都只会放大错误。

## 九、顺序是否正确的判断标准

实施顺序是否正确，不看页面做得多快，也不看接口数量，而看以下四件事是否成立：

1. `trend` 能独立构建 baseline world，而不是历史报表重述
2. `simulation` 能独立接收结构化政策情景，并在 baseline 上做反事实比较
3. 所有结论都能回到真实项目事实、统一口径和明确假设
4. analysis mart 始终只是分析层宽表，而不是整个系统的业务本体

只要这四点成立，实施顺序就是对的；只要有任一点不成立，系统就还没有进入真正的 `sandbox` 轨道。
