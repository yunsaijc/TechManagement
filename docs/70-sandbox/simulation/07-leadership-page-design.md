# 领导页展示设计

## 一、这份文档解决什么问题

这份文档只回答一件事：

`simulation 的领导页应该怎么展示`

它不讨论接口命名，不讨论引擎细节，只讨论：

- 页面要讲什么
- 页面按什么顺序讲
- 现有数据真正能支撑到哪一层

## 二、页面的核心任务

领导页不是“结果堆叠页”，而是要顺着领导的阅读顺序，回答四个问题：

1. 这次怎么调
2. 总盘子怎么变
3. 哪些对象变得最多
4. 影响是怎么传过去的

## 三、标准页面结构

页面固定为六个区块。

### 1. 左侧控制区

作用：回答 `这次准备怎么调`

应包含：

- 调整对象
- 调整方式
- 生效时间
- 运行推演

页面上必须使用人话表达，例如：

- `对基础研究专项（自然科学基金）/ 1010101-面上项目增加支持`
- `新增经费 300 万元`
- `新增项目 10 个`

不能直接展示：

- `funding_boost = 0.65`
- 内部 action id
- 原始字段名

### 2. 顶部总览卡

作用：回答 `整体会变成什么样`

建议固定为五张卡：

- 总申报项目数
- 总立项项目数
- 总经费
- 平均资助强度
- 影响主题数

所有卡片都应是 `baseline vs scenario` 的差值表达。

### 3. 中部三张结果图

作用：回答 `哪些对象变化最大`

建议固定为：

- 申报项目数变化 TOP10
- 立项项目数变化 TOP10
- 经费分布变化

### 4. 左下传导图

作用：回答 `变化怎么传过去`

主图应以 `topic` 为主对象：

- `node = topic`
- `edge = topic -> topic`
- 实线表示直接影响
- 虚线表示传导影响

项目、政策、文档不应全部塞进主图做 node。

### 5. 中下关键结果表

作用：回答 `最值得领导看的变化有哪些`

建议列为：

- 影响类型
- 影响对象
- 影响方向
- 变化幅度
- 置信度

### 6. 右下说明区

作用：回答 `这页结论应该怎么理解`

至少包含两块：

- 推演说明
- 置信度说明

## 四、当前数据下真正能展示什么

页面必须服从真实数据边界，不能为了界面好看而虚构粒度。

### 1. 当前可以稳定展示的粒度

- `专项`
- `指南项`
- `年份`
- `申报变化`
- `立项变化`
- `经费变化`
- `直接影响 / 传导影响`

因此第一版领导页的主展示对象应统一为：

- `专项 / 指南项`

### 2. 当前不应假装已经稳定支持的粒度

- 更细的真实学科方向名
- 直接从 raw 政策文本生成正式方案原文
- 项目级大规模主图

如果当前没有稳定的方向名，就应老老实实展示到 `专项 / 指南项`，不能把多个不同对象都写成一堆“面上项目”。

## 五、页面表达纪律

- 不展示内部字段名
- 不展示玩具式参数
- 不重复讲同一层信息
- 不把 `0 经费 + 0 项目` 的对象混进“本次怎么调”
- 不把代理结果说成硬事实

## 六、和输出包的关系

为了稳定支撑这张领导页，后端输出层最终应补齐一层页面合同。

建议页面直接消费如下结构：

```json
{
  "control_panel": {},
  "summary_cards": [],
  "application_top10": [],
  "funded_top10": [],
  "funding_distribution": {},
  "propagation_graph": {},
  "impact_table": [],
  "narrative": [],
  "confidence": []
}
```

这层是展示合同，不替代 simulation 的正式输出包。

正式输出包仍按 [04-outputs-and-api.md](/home/tdkx/workspace/tech/docs/70-sandbox/simulation/04-outputs-and-api.md) 定义。
