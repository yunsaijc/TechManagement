# 沙盘推演总览

## 一、定位

沙盘推演不是“调参数看图动一下”，而是在 `Baseline World` 上施加结构化政策动作后的反事实比较。

它回答的问题是：

- 如果调整指南、准入、评审、配额、预算、协作支持等政策，未来会怎样不同
- 谁受益，谁受损，谁被挤出，代价是什么

## 二、正式定义

一套完整沙盘应定义为：

`Sandbox = Baseline World + Policy Package + Constraints + Evaluation Goals + Assumptions`

这五块缺一不可。

## 三、推演对象

沙盘推演的对象不是单个项目成败，而是：

- 主题结构
- 立项结构
- 资金结构
- 机构分布
- 人员结构代理
- 风险暴露
- 资源挪腾代价

本质上，沙盘推演的是：

`政策动作如何改变省级科研投资组合`

## 四、输入

推演输入必须是一份 `Scenario Contract`，至少包括：

- `intent`
- `baseline_scope`
- `actions`
- `constraints`
- `evaluation`
- `validation`

不允许再把一组零散临时参数当成正式政策输入。

## 五、状态转移

所有政策效果都必须沿真实治理流程走：

1. `申报响应`
   政策先影响谁会来报、报多少、报多大。
2. `评审选择`
   再影响谁过线、谁入选、谁被挤出。
3. `合同配置`
   再影响合同与经费如何落地。
4. `结构外溢`
   最后才影响协作、迁移和主题结构。

任何跳过中间环节、直接修改结果变量的做法，都不能称为正式沙盘。

## 六、输出

沙盘推演必须输出：

- `baseline`
- `scenario`
- `counterfactual_comparison`
- `winners_and_losers`
- `crowding_out`
- `risk_shift`
- `evidence_backed_recommendations`

## 七、当前数据下的动作支持边界

当前数据条件下，最适合进入第一版主流程的动作是：

- 预算重分配
- 配额调整
- 优先级调整
- 评审阈值调整

当前不应假装已完全支持的动作包括：

- 验收规则调整
- 成果转化激励
- 区域差异化政策
- 真实人才支持效果

## 八、LLM 在 Simulation 中的职责

LLM 用于：

- 领导自然语言政策意图解析
- 政策动作结构化
- 推演结果归纳与解释
- 多方案比较的文字组织

LLM 不用于：

- 替代引擎计算状态转移
- 替代预算/配额/评审规则计算
- 把代理效果说成强因果

## 九、与 Trend 的关系

Trend 负责形成 baseline；
Simulation 负责在 baseline 上施加政策动作。

两者共用：

- 数据底座
- 对象模型
- 指标口径
- 证据边界
- LLM 交互层
