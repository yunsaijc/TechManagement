# 趋势预判分析链路

## 一、总目标

趋势预判的分析链路必须服务于一个目标：

在当前政策不变的前提下，基于真实项目数据和图谱结构，构建 `Baseline World`，并输出对未来 1-2 个年度周期的条件性判断。

这条链路不是“做个热点图”，而是从事实层、关系层走到统计层、信号层，再走到基线预测层。

## 二、总链路

```text
项目业务库 + 图谱 + 外部辅助信号
  -> 事实层重建
  -> 关系层统一
  -> 统计层构建
  -> 变化信号识别
  -> 基线世界预测
  -> 风险与机会排序
  -> 领导简报与证据链输出
```

其中：

- 项目业务库和图谱是主底座
- 外部信息只能走辅助信号支路，不能改写主链路事实

## 三、阶段设计

### 1. 事实层重建

先把项目生命周期事实补齐到可分析状态。

核心工作：

- 用 `Sb_Jbxx.id` 统一项目主键
- 串起 `Sb_Jbxx -> Sb_Sbzt -> Sb_Jfgs -> PS_XMPSXX -> Ht_XMLXXX -> Ht_Jbxx -> Ht_Jfgs`
- 区分申报、评审、立项、合同、经费等事件时间
- 明确哪些字段是直接观测，哪些只是后续派生

输出结果：

- `Project`
- `Review`
- `Funding`
- `Contract`

如果这一层不稳定，后面所有趋势判断都会失真。

### 2. 关系层统一

这一层解决“结构怎么看”的问题。

核心工作：

- 统一 `Topic` 口径，优先使用指南方向
- 统一项目与机构、人员、主题的映射关系
- 将图谱中的结构关系与项目事实挂接到同一主题体系下
- 明确图谱指标是结构代理，而不是业务事实

输出结果：

- `Project -> Topic`
- `Project -> Institution`
- `Project -> Person`
- `Topic <-> Topic`
- `Institution <-> Institution`

这一层的目标不是画图，而是让“热点迁移”“协作变化”“主导格局变化”有坚实结构基础。

### 3. 统计层构建

统计层负责把实体和关系转成可计算、可对比的分析状态。

核心工作：

- 构建 `topic_year_state`
- 构建 `topic_institution_year_state`
- 构建 `topic_person_year_state`
- 构建 `project_cohort_outcome`
- 构建 `portfolio_year_state`

这里的 `topic × year` 只是主分析宽表，不是业务世界观。

它承担的职责是：

- 看当前布局
- 看结构变化
- 支撑短期基线预测

### 4. 变化信号识别

这一层负责识别“已经发生的变化”，而不是直接输出判断口号。

核心工作：

- 识别持续升温、持续回落主题
- 识别经费与热度背离
- 识别机构集中、主体流失、协作衰减
- 识别热点迁移通道
- 识别需要进入领导关注名单的异常主题

输出结果是结构化 `signals`，例如：

- 申报量连续增长
- 合同经费增速转负
- 新进入机构占比下降
- 相邻主题持续吸走资源

每条 `signal` 都必须回到具体指标和时间窗。

### 5. 基线世界预测

这是趋势模块与普通统计模块真正拉开的地方。

它回答：

- 如果当前政策不变，哪些趋势会延续
- 哪些结构矛盾会继续扩大
- 哪些主题下一个周期更可能升温、回落或积累风险

形成方式不是让 LLM 直接写结论，而是：

- 用历史状态和变化信号形成趋势方向
- 用图谱结构判断迁移与联动
- 用组合层约束判断资源挤占与结构延续
- 必要时用外部辅助信号做解释增强或排序校正

输出结果：

- `baseline_forecast`
- `risk_alerts`
- `opportunity_ranking`

### 6. 领导表达层生成

最后一层不是“重新分析”，而是把结构化结果组织成领导能看懂的页面与问答。

应输出四类内容：

- 看现在：当前布局和资源结构
- 看变化：近几年发生了什么
- 看未来：基线世界会怎么走
- 看风险：哪些地方需要提前管

这一步允许使用 LLM，但 LLM 只负责：

- 归纳表达
- 追问展开
- 证据链组织

不能替代底层计算。

## 四、外部辅助信号支路

外部信息不能进主事实层，但可以作为辅助信号支路存在。

使用方式：

```text
政策文本 / 新闻 / 论文 / 专利 / 产业事件
  -> topic 对齐
  -> 形成 external_signal
  -> 参与解释、排序校正、风险提示
```

使用边界：

- 不可改写项目数、经费数、立项数等主状态
- 不可单独构成“领导硬结论”
- 必须保留来源、时间、覆盖度、可信度

## 五、证据与表达规则

趋势链路中的每个输出对象都必须带证据说明：

| 输出类型 | 最低要求 |
|---|---|
| `snapshot` | 给出指标值、口径、时间窗 |
| `signal` | 给出触发指标、变化窗口、证据类型 |
| `forecast` | 给出基线假设、证据窗口、不确定性 |
| `risk` | 给出风险机制、支撑指标、边界声明 |

不允许：

- 直接把相关性说成因果
- 直接把代理指标说成真实结果
- 直接把辅助信号说成主事实

## 六、实现上的关键要求

趋势链路在实现上必须遵守以下要求：

1. 先统一实体和关系，再做宽表
2. 先识别已发生变化，再做未来判断
3. 先形成结构化结果，再让 LLM 生成表达
4. 所有预测都是 `baseline world` 下的条件性判断

因此，趋势模块的核心不是一个图，也不是一个排序，而是一条从真实数据到基线未来的完整分析链。
