# LogicOn 规则设计

## 规则设计原则

- 规则优先：可用数学/时间确定比对的问题优先用规则解决
- 证据驱动：规则输出必须包含证据定位，便于复核
- 分级输出：证据充分且可确定的输出 RED；证据不足但存在异常信号的输出 YELLOW
- 最小误报：遇到抽取不全时不强行下结论，避免“解析失败”被误判为冲突

## 规则清单（最小可用集合）

### R-TIME-01 执行期与进度安排跨度冲突

目的：发现“整体执行期”与“详细任务进度安排/里程碑”跨度不一致。

输入：

- 执行期：起止年月/总年限（来自基本信息或摘要表）
- 进度节点：里程碑日期、年度/阶段目标年份集合（来自进度安排章节）

判定逻辑（示例）：

- 若执行期为 `start..end`，且里程碑最晚日期 `> end + date_tolerance_days`，输出 RED
- 若仅能抽到“执行期=2年”，但里程碑年份集合跨度 ≥ 3 年，输出 YELLOW（建议复核）

输出建议：

- title: `执行期与进度跨度不一致`
- description: 指出执行期与最晚里程碑/年份跨度，并列出相关证据片段

### R-BUDGET-01 预算总额与明细求和不一致

目的：发现“预算总额/资金申请总额/下达总额”与明细求和不一致。

输入：

- 总额：资金申请/下达总额、项目总预算（来自预算汇总表或摘要字段）
- 明细：预算科目明细（来自预算表/预算明细表）

判定逻辑（示例）：

- 计算 `sum(items)` 与 `total` 的差值
- 若差值绝对值超过阈值（例如 `amount_tolerance`），输出 RED
- 若总额抽取不到，但抽到多处“总额”且互相不一致，输出 YELLOW

### R-BUDGET-02 单位预算汇总不一致（承担单位/合作单位）

目的：发现“承担单位、合作单位经费预算明细表”内部或与单位合计行的不一致。

输入：

- 单位预算明细：单位 + 科目金额
- 单位合计（若有）：单位合计/小计

判定逻辑：

- 对每个单位按科目求和，与该单位合计比对
- 明确不一致输出 RED

### R-METRIC-01 同一指标多处目标值不一致

目的：发现同一指标在不同章节出现不同目标值/单位/约束。

输入：

- 指标实体：`metric_name`、`value`、`unit`、`constraint`
- 来源章节：绩效目标表、总体目标/实施期目标、验收考核指标等

判定逻辑（示例）：

- 通过同义归一化（字符串归一 + 可选 LLM）将指标聚类
- 同一簇内存在不同目标值（且差异超过容忍区间）→ RED
- 同一簇内单位不一致但可换算不确定 → YELLOW

### R-PERSON-01 人员角色与分工表冲突（可选）

目的：发现同一人员在不同章节角色/分工描述冲突（例如负责人/联系人混淆）。

判定逻辑：

- 抽取“项目负责人/联系人/项目组成员”集合
- 若同名人员在“非成员表字段”与“成员表字段”角色冲突，输出 YELLOW

## 风险分级口径

- RED：可确定的数值或时间冲突
- YELLOW：存在冲突信号但证据不足或需要语义复核
- GREEN：未发现冲突（或规则不适用）

## 规则可配置项

- 金额容忍：`amount_tolerance`（默认 0.01 万元）
- 时间容忍：`date_tolerance_days`（默认 30 天）
- 指标容忍：`metric_tolerance_ratio`（默认 0.01，用于浮点误差或口径差）
