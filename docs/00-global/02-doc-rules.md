# 📚 文档维护规范

## 概述

本文档定义了科技管理系统项目的文档维护规范，确保文档结构统一、内容同步，便于团队协作和维护。

## 文档目录结构

### 全局文档 (00-global/)

```
00-global/
├── 01-environment.md    # 环境配置
├── 02-doc-rules.md      # 文档维护规范（本规范）
├── 03-coding-rules.md   # 开发规范
└── 04-deployment.md     # 部署文档
```

### 通用组件文档 (10-common/)

```
10-common/
├── 01-overview.md       # 通用组件概述
├── 02-models.md         # 通用数据模型
├── 03-llm.md           # LLM 统一封装
├── 04-file-handler.md   # 文件处理
├── 05-vision.md         # 视觉能力
└── 06-tools.md          # 工具函数
```

### 服务文档 (XX-service/)

每个服务目录必须遵循统一的文档结构：

```
20-review/                       # 服务目录（编号-服务名）
├── 01-overview.md              # 服务概述（必填）
├── 02-rules.md                 # 规则设计（可选）
├── 03-agent.md                # Agent 设计（可选）
├── 04-document-parser.md      # 文档解析方案（可选）
└── 05-api.md                  # API 接口文档（必填）
```

#### 文档编号规范

| 编号 | 文档 | 必填 | 说明 |
|------|------|------|------|
| `01-` | overview | ✅ | 服务概述 |
| `02-` | rules | ○ | 规则设计（适用于有规则引擎的服务） |
| `03-` | agent | ○ | Agent 设计（适用于有 Agent 的服务） |
| `04-` | xxx-impl | ○ | 实现细节/解析方案等 |
| `05-` | api | ✅ | API 接口文档 |

> ○ = 可选，根据服务复杂度决定是否创建

## 文档命名规范

### 文件命名

- 使用中文命名，清晰表达内容
- 文件名格式：`XX-功能描述.md`（如 `02-rules.md`）
- 避免特殊字符、空格、emoji

### 目录命名

- 格式：`XX-name/`
- 编号 + 语义名称（如 `20-review/`）

## 文档内容要求

### 1. 概述文档 (01-overview.md)

必须包含：
- **服务定位**：是什么、做什么、解决什么问题
- **业务流程图**：用 ASCII 图或 Mermaid 展示流程
- **核心技术选型**：使用的技术和框架
- **上下游依赖关系**：依赖哪些模块/服务

### 2. API 文档 (05-api.md)

必须包含：
- 接口列表（路径、方法、参数）
- 请求/响应示例（JSON 格式）
- 错误码说明

### 3. 设计文档 (02/03-xxx.md)

必须包含：
- 设计思路（为什么这么设计）
- 核心代码结构（类/函数）
- 使用示例（可运行的代码片段）

## 文档与代码同步

| 操作 | 文档更新 |
|------|----------|
| 新增服务 | 创建服务文档，更新 `docs/index.md` |
| 新增功能 | 更新对应文档 |
| API 变更 | 更新 `05-api.md` |
| 重构代码 | 同步更新设计文档 |
| 依赖变更 | 更新 `01-environment.md` |

## 全局文档更新规则

| 文档 | 更新时机 |
|------|----------|
| `index.md` | 新增/删除服务目录 |
| `01-environment.md` | 依赖变更、环境配置变化 |
| `02-doc-rules.md` | 文档规范调整（本规范） |
| `03-coding-rules.md` | 开发规范变化 |
| `04-deployment.md` | 部署方式变化 |

## 文档目录索引维护

每次新增服务后，必须更新 `docs/index.md`：

1. **目录树**：在对应位置添加新服务目录
2. **层次关系图**：添加新服务节点
3. **快速导航**：添加文档链接
4. **服务依赖关系**：更新依赖图

### 更新示例

```markdown
## 新增 30-project 服务后的 index.md 更新

### 目录结构添加：
├── 30-project/                   # 📋 项目评审服务
│   ├── 01-overview.md           # 服务概述
│   └── 05-api.md                # API 接口文档

### 层次关系添加：
├── 30-project/         ← 项目评审服务（未来，依赖 common）
```

## 文档审核清单

提交文档前检查：

- [ ] 链接有效性（内部链接可跳转）
- [ ] 代码示例语法正确
- [ ] 术语一致性（如统一使用"审查"而非混用"审核"）
- [ ] 无错别字
- [ ] 图片路径正确

## 文档格式规范

### 标题层级

```markdown
# 一级标题（文档标题）
## 二级标题（章节）
### 三级标题（子章节）
#### 四级标题（小节）
```

### 代码块

- 使用语言标签：` ```python `
- 代码保持缩进一致
- 避免超长代码行（建议 ≤ 120 字符）

### 表格

- 使用 Markdown 表格格式
- 表头对齐
- 内容简洁

### 链接

- 内部链接使用相对路径：`[文档名称](./path/doc.md)`
- 外部链接使用完整 URL
