# 📋 分组子服务设计

## 概述

分组子服务负责将申报项目按“项目在做什么”进行分组。当前实现采用**学科代码优先、项目语义辅助**的策略：先按学科代码体系和层级做粗分，再对超大桶使用项目名称 `xmmc` 和项目简介 `xmjj` 做语义拆分。

分组的基本原则是：
- 同代码项目可以拆到不同组
- 不同代码项目可以合到同一组
- 组内主题必须尽量一致，且能被人类评审解释
- 小于 `min_per_group` 的组不能原样保留，必须继续归并或拆散处理

---

## 核心逻辑

### 分组目标

1. 先根据学科代码体系与层级做粗分
2. 再对超大桶根据项目名称和可选简介提炼主题语义
3. 使用 embedding / 聚类对大桶进行二次拆分
4. 再根据容量约束对过小组和过大组进行修正
5. 最终输出可解释、可回放的分组结果

### 不再使用的规则

以下逻辑不再作为主分组依据：
- 质量评分驱动的均衡分配
- 完全脱离学科代码的纯文本语义分组
- 把小组整体移动到一个目标组

学科代码仍可作为辅助信息：
- 用于粗分和层级相似性判断
- 用于识别字母代码体系与 7 位数字体系
- 不能作为同组绝对硬约束

### 字母代码与数字代码隔离

字母开头学科代码与 7 位数字学科代码属于两套体系：
- 两套体系不能混在一个组里
- 同体系内可以通过语义判断拆分或合并
- 若语义非常接近但跨体系，默认仍应分开，除非业务明确允许合并并人工复核

---

## 推荐流程

```
1. 获取项目列表
2. 读取学科代码并按代码体系/层级做粗分
3. 找出超过 `max_per_group` 的大桶
4. 仅对大桶提取项目语义（`xmmc` 主、`xmjj` 辅）
5. 用 embedding / 聚类对大桶做二次拆分
6. 对小于 min_per_group 的组逐项目重分配
7. 对超过 max_per_group 的组继续拆分
8. 按需生成组标题、理由和复核标记
9. 做字母代码 / 数字代码体系隔离检查
10. 输出分组结果 + 理由 + 复核标记
```

---

## 详细策略

### 1. 语义提取

对每个项目先生成内部语义表示：
- 研究对象
- 研究问题
- 方法/技术路线
- 应用场景
- 核心关键词

这一步优先只看 `xmmc`，当 `xmjj` 存在时再补充。

### 2. 学科粗分

分组第一步先利用学科代码形成粗粒度桶：

- 优先区分字母代码体系与数字代码体系
- 在同一体系内按代码前缀和层级相似性组织项目
- 这一阶段只解决“先和谁放在一起看”的问题，不直接输出最终组

### 3. 大桶语义拆分

为了控制成本和提升速度，仅对超大桶使用语义能力：

- 计算桶内项目文本的 embedding
- 基于相似度进行聚类或切分
- 必要时再调用 LLM 生成组标题或补充解释

这样可以避免对全部项目逐条调用 LLM。

### 4. 语义分组规则

语义层的任务不是替代学科代码，而是在代码主线之上模拟人类评审：
- 看项目到底在研究什么
- 看方法是否一致
- 看应用目标是否一致
- 看是否属于同一主题簇

允许以下情况：
- 同代码但主题不同 → 拆开
- 不同代码但主题相近 → 合并
- 无法明确归类 → 标记 `needs_review`

### 5. 小组处理

对于小于 `min_per_group` 的组，不能只找一个目标组整体迁移，而要逐项目处理：
- 每个项目单独找最合适的组
- 一个小组里的项目可以分散到多个组
- 如果某个项目找不到合适目标组，保留待复核

### 6. 大组处理

对于超过 `max_per_group` 的组：
- 优先按语义切分成多个子主题
- 组标题优先采用规则生成，必要时再由 LLM 命名
- 每个子主题尽量保持内部一致

### 7. 体系隔离检查

分组结束后再做一次硬检查：
- 字母代码组里不能混入数字代码项目
- 数字代码组里不能混入字母代码项目
- 如已混入，则回退该轮合并结果并重新分配

---

## 结果要求

每个组至少输出：
- `group_name`
- `group_reason`
- `count`
- `needs_review`
- 组内每个项目的 `project_reason`
- 每个项目的 `confidence`

最终结果要能支持：
- 人工查看
- 追溯为什么这样分
- 重新运行时结果稳定

## 伪代码

```python
projects = load_projects(year)
semantics = build_semantics(projects)
clusters = semantic_cluster(projects, semantics)
clusters = llm_refine_groups(clusters)
clusters = redistribute_small_groups(clusters, min_per_group)
clusters = split_large_groups(clusters, max_per_group)
clusters = enforce_code_system_separation(clusters)
return export_result(clusters)
```

---

## 约束参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `min_per_group` | 5 | 小组下限，低于则继续归并 |
| `max_per_group` | 15 | 每组上限，超过则拆分 |
| `top_k_candidates` | 3 | 每个项目保留的候选组数 |
| `confidence_threshold` | 0.65 | 低于该值标记复核 |
| `enable_embedding` | true | 是否启用 embedding 召回 |
| `enable_llm` | true | 是否启用 LLM 复核 |

---

## 运行建议

- 默认以学科代码为主线，`xmmc` 为主，`xmjj` 为辅
- 优先只对超大桶做 embedding，减少成本
- 所有低置信度项目都要保留复核信息
- 不要把学科代码当绝对硬约束，也不要完全脱离代码
