# 📋 分组子服务设计

## 概述

分组子服务负责将申报项目按“项目在做什么”进行分组。它不再把学科代码当作硬约束，也不再依赖质量评价；核心输入只有项目名称 `xmmc`，项目简介 `xmjj` 作为可选补充。

分组的基本原则是：
- 同代码项目可以拆到不同组
- 不同代码项目可以合到同一组
- 组内主题必须尽量一致，且能被人类评审解释
- 小于 `min_per_group` 的组不能原样保留，必须继续归并或拆散处理

---

## 核心逻辑

### 分组目标

1. 先根据项目名称和可选简介提炼主题语义
2. 再用 LLM 按“研究对象、方法、应用场景、目标”做分组
3. 再根据容量约束对过小组和过大组进行修正
4. 最终输出可解释、可回放的分组结果

### 不再使用的规则

以下逻辑不再作为主分组依据：
- 仅按学科代码前缀分组
- 质量评分驱动的均衡分配
- 把小组整体移动到一个目标组

学科代码仍可作为辅助信息：
- 用于提示主题边界
- 用于识别字母代码体系与 7 位数字体系
- 不能作为同组硬约束

### 字母代码与数字代码隔离

字母开头学科代码与 7 位数字学科代码属于两套体系：
- 两套体系不能混在一个组里
- 同体系内可以通过语义判断拆分或合并
- 若语义非常接近但跨体系，默认仍应分开，除非业务明确允许合并并人工复核

---

## 推荐流程

```
1. 获取项目列表
2. 提取项目语义（仅 xmmc，xmjj 可选）
3. embedding 召回候选组（可选）
4. LLM 判断主题归属
5. 形成初始分组
6. 对小于 min_per_group 的组逐项目重分配
7. 对超过 max_per_group 的组拆分
8. 做字母代码 / 数字代码体系隔离检查
9. 输出分组结果 + 理由 + 复核标记
```

---

## 详细策略

### 1. 语义提取

对每个项目先生成内部语义表示：
- 研究对象
- 研究问题
- 方法/技术路线
- 应用场景
- 核心关键词

这一步优先只看 `xmmc`，当 `xmjj` 存在时再补充。

### 2. 候选组召回

为了降低 LLM 成本，可以先用 embedding 做候选召回：
- 计算项目与已有组主题中心的相似度
- 取 top-k 候选组
- 再交给 LLM 判断最终归属

如果没有 embedding，也可以直接让 LLM 做全量判断，但成本更高。

### 3. LLM 分组规则

LLM 的任务不是按代码分类，而是模拟人类评审：
- 看项目到底在研究什么
- 看方法是否一致
- 看应用目标是否一致
- 看是否属于同一主题簇

允许以下情况：
- 同代码但主题不同 → 拆开
- 不同代码但主题相近 → 合并
- 无法明确归类 → 标记 `needs_review`

### 4. 小组处理

对于小于 `min_per_group` 的组，不能只找一个目标组整体迁移，而要逐项目处理：
- 每个项目单独找最合适的组
- 一个小组里的项目可以分散到多个组
- 如果某个项目找不到合适目标组，保留待复核

### 5. 大组处理

对于超过 `max_per_group` 的组：
- 优先按语义切分成多个子主题
- 再由 LLM 给每个子主题命名
- 每个子主题尽量保持内部一致

### 6. 体系隔离检查

分组结束后再做一次硬检查：
- 字母代码组里不能混入数字代码项目
- 数字代码组里不能混入字母代码项目
- 如已混入，则回退该轮合并结果并重新分配

---

## 结果要求

每个组至少输出：
- `group_name`
- `group_reason`
- `count`
- `needs_review`
- 组内每个项目的 `project_reason`
- 每个项目的 `confidence`

最终结果要能支持：
- 人工查看
- 追溯为什么这样分
- 重新运行时结果稳定

## 伪代码

```python
projects = load_projects(year)
semantics = build_semantics(projects)
clusters = semantic_cluster(projects, semantics)
clusters = llm_refine_groups(clusters)
clusters = redistribute_small_groups(clusters, min_per_group)
clusters = split_large_groups(clusters, max_per_group)
clusters = enforce_code_system_separation(clusters)
return export_result(clusters)
```

---

## 约束参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `min_per_group` | 5 | 小组下限，低于则继续归并 |
| `max_per_group` | 15 | 每组上限，超过则拆分 |
| `top_k_candidates` | 3 | 每个项目保留的候选组数 |
| `confidence_threshold` | 0.65 | 低于该值标记复核 |
| `enable_embedding` | true | 是否启用 embedding 召回 |
| `enable_llm` | true | 是否启用 LLM 复核 |

---

## 运行建议

- 默认以 `xmmc` 为主，`xmjj` 为辅
- 先做 embedding 再做 LLM，减少成本
- 所有低置信度项目都要保留复核信息
- 不要把“代码体系”当成分组主线
